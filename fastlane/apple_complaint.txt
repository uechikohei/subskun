ISSUE 2: GUIDELINE 5.1.1(v) - ACCOUNT DELETION

The reviewer stated: "The app supports account creation but does not include an option to initiate account deletion."

THIS IS FACTUALLY INCORRECT. Account deletion is FULLY IMPLEMENTED and has been present in EVERY build submitted for review. The reviewer simply did not find it.

EXACT NAVIGATION PATH TO ACCOUNT DELETION:

1. Launch the app and sign in with a Google or Apple account.
2. Tap the "Settings" tab in the bottom tab bar (gear icon, rightmost tab).
3. Tap the "Account" row. It displays the user's name and a chevron icon.
4. A modal sheet opens showing the Account screen.
5. The screen displays: login method, display name, email address, a "Sign Out" button, and below it, a RED "Delete Account" button in destructive style.
6. Tap the RED "Delete Account" button.
7. A confirmation dialog appears with the title "Delete your account?" and the message "All subscription data and settings will be permanently deleted. This action cannot be undone."
8. Tap "Delete Account" to confirm.
9. The app performs the following: disconnects Google Sign-In or revokes Apple authentication token, deletes ALL subscription data from the local database, clears ALL user settings and cached data from UserDefaults, resets app settings to defaults, and returns the user to the sign-in screen.

This feature is implemented in:
- UI: SubscriptionTracker/Features/Auth/AccountView.swift (lines 44-100)
- Logic: SubscriptionTracker/Features/Auth/AuthenticationStore.swift (deleteAccount method, lines 128-159)
- Apple token revocation: AuthenticationStore.swift (revokeAppleToken method, lines 445-468)
- Tests: SubscriptionTrackerTests/AccountDeletionTests.swift

The button is RED, clearly labeled "Delete Account" (or "アカウントを削除" in Japanese), and placed in a standard location that follows iOS design conventions. It takes a reviewer FIVE TAPS to verify this feature exists. The fact that it was missed raises serious concerns about the quality and thoroughness of the review.

I have addressed this exact issue in a previous submission cycle. The feature was already implemented at that time as well. It continues to be rejected for the same incorrect reason.
